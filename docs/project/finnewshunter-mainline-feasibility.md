# FinnewsHunter 主线可行性判断

## 结论

主线可以继续坚持，而且对我们这次作业来说是更合适的路线。

但要说得准确一点：

- **可以完成整条研究链路**
- **不建议把仓库里所有中文站点都视为同等可用**

更现实的主线应该是“4 个核心源 + 1 到 2 个扩展源”：

1. 核心源：`CNStock / JRJ / NBD / Sina Finance`
2. 扩展源：`Yicai / Caixin`
3. `STCN` 暂时不作为主抓取源

## 为什么这么判断

从 `FinnewsHunter` 的仓库说明可以确认：

- 它的旧版项目就是做中文财经历史新闻爬取
- 明确抓取的网站包括：
  - 新浪财经
  - 每经网
  - 金融界
  - 中国证券网
  - 证券时报
- 旧版说明里写得很清楚，历史新闻会存到 `MongoDB`
- 单条新闻的核心字段包括：
  - 时间
  - 网址
  - 标题
  - 正文

更重要的是，它后续更新日志把“哪些站点仍然适合作为主线”说得比较明白：

- `cnstock`：有明确的历史抓取和实时更新示例
- `jrj`：有明确的历史抓取示例
- `nbd`：在后续构建股票新闻库的例子里持续被使用
- `sina`：说明里标注了“未更新”
- `stcn`：说明里明确写了旧代码已不可用

## 对我们项目意味着什么

这说明我们不该把主线理解成：

- “直接无脑跑整个 FinnewsHunter 仓库”

而应该理解成：

- “沿用 FinnewsHunter 这条中文历史抓取路线，但优先选它里面仍然现实可用的站点”

## 我们现在的主线定义

为了和仓库现实保持一致，同时满足“4 到 6 个主要站点就够”的目标，当前项目里的主线优先级改成了：

1. `CNStock`
2. `JRJ`
3. `NBD`
4. `Sina Finance`
5. `Yicai`
6. `Caixin`
7. `STCN` 关闭

## 能不能完成我们全部调用

如果“全部调用”指的是整条研究流程：

- 历史新闻获取
- 结构化存储
- Qwen 五维打分
- 周特征构建
- 周频训练与简单策略

那么答案是：

- **可以**

因为我们真正需要的是一批稳定、质量够好、数量也够大的中文历史新闻样本，不是必须把仓库里每一个站点都跑通。

如果“全部调用”指的是“把 FinnewsHunter 里所有中文源都照搬并逐个跑通”，那答案就不是完全肯定：

- `STCN` 不适合继续当主线
- `Sina` 需要谨慎看待

## 推荐执行法

接下来主线应当这样推进：

1. 先围绕 `CNStock / JRJ / NBD / Sina Finance` 这 4 个核心源确认能不能拿到一批真实历史输出
2. 输出后直接导入我们现在已经写好的主流程
3. 如果 4 个核心源样本仍不够，再尝试 `Yicai / Caixin`
4. 只有 6 个主要源合起来仍不够，才考虑备线补样本
